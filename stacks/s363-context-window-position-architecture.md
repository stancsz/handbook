# S-363 · Context Window Position Architecture

The five-layer production agent architecture (perception → reasoning → planning → tool integration → memory) is well-understood. What's less appreciated is that *where* each layer's output sits in the context window determines whether the model uses it. Context position — boundaries vs. middle, ordering within layers, KV-cache-friendly prefixing — is an architectural decision, not an implementation detail. Get it right and a mid-tier model outperforms a frontier model with a sloppy window.

## Forces

- **The "lost in the middle" problem is structural, not fixable with prompting.** Models attend disproportionately to the first and last tokens of context windows. Content in the middle — verbose tool schemas, irrelevant history, filler retrieved chunks — degrades the model's ability to reason about the task, regardless of model tier.
- **Context assembly happens at inference time, not at design time.** A production context pipeline has five dynamically assembled layers: (1) system instructions, (2) retrieved knowledge filtered for relevance, (3) persistent memory, (4) compressed conversation history, (5) tool definitions. The order and positioning of these layers at each inference step is a runtime decision, not a static prompt.
- **Naive RAG retrieval amplifies middle-position failures.** When top-k retrieval dumps 10 chunks into context, the most relevant one is often in the middle of the resulting prompt — exactly where the model is least likely to weight it. Hybrid search with RRF and reranking can surface the right chunk but fail to position it correctly.
- **KV cache state is path-dependent across turns.** Long-running agents accumulate cache state that breaks if you reorder context elements between turns. KV-cache-friendly prefix structures — where critical information is pinned to the beginning of each new turn's prefix — are a prerequisite for stable multi-step reasoning.
- **88% of AI agent projects never reach production** — enterprise analyses consistently find this gap. The root cause is rarely the model. It's context rot: the slow accumulation of noise, redundancy, and misaligned attention that degrades every output the model produces without triggering errors.

## The Move

Design the context pipeline as a first-class architectural layer with explicit position contracts:

- **Anchor critical signals at context boundaries.** System instructions and the most task-relevant context belong at the start and end of the context window — not buried in the middle. If a retrieved chunk is the single most important piece of context for the current step, ensure it appears at the beginning of the prompt, not in the middle after verbose tool schemas.
- **Place tool definitions at the end, not the beginning.** Tool schemas are reference material — the model reads them to know what actions are available, then acts. If they're at the top of the context, they consume the high-attention boundary zone with low-signal content. Move them to the end or use structured references that expand only when needed.
- **Filter retrieved context to minimum-necessary before insertion.** Don't dump full top-k chunks. Run relevance scoring, extract only the sentences that directly address the query, and place those at boundaries. One highly relevant sentence at the start beats ten relevant chunks buried in the middle.
- **Design for cache-friendly prefixing at each turn.** In long-running agents, the KV cache persists across turns. If critical information (task goal, current progress, constraints) is in a middle position at turn N, it becomes a middle position in the cached prefix at turn N+1. Pin goal state and constraint signals to turn-start positions in the prefix structure so they remain in the high-attention zone across turns.
- **Compress conversation history proactively, not lazily.** By turn 10 of a multi-turn agent, full history is toxic — it buries the current task under noise and increases token cost without improving output. Compress history into a summary of decisions made and current state at each turn boundary, before the context grows past the model's effective attention window (typically 32k tokens for reliable multi-step reasoning even in 1M-token models).
- **Separate perception from reasoning in context structure.** Document parsing, multimodal preprocessing, and multimodal content extraction belong in a distinct "perception" layer that feeds cleaned, filtered context into the reasoning layer — not raw document dumps. What you *exclude* matters as much as what you include.

## Evidence

- **Blog post (Tian Pan, tianpan.co, Oct 2025):** One company shipped 7,949 AI agents; 15% worked. The 88% failure rate in enterprise agent projects traces to architecture problems, not model problems. A production context pipeline has five dynamically assembled layers at inference time, and context position directly determines whether the model attends to it. — [tianpan.co](https://tianpan.co/blog/2025-10-23-ai-agent-architecture-production)
- **Blog post (Tian Pan, tianpan.co, Feb 2026):** The fundamental problem is "context rot" — accumulation of noise and misaligned attention. Treating context design as a first-class architectural concern (not a debugging-phase prompt tweak) is the core of context engineering. Every major model — GPT-4.1, Claude Opus 4, Gemini 2.5 — exhibits attention degradation in the middle of long contexts without exception. — [tianpan.co](https://tianpan.co/blog/2026-02-06-context-engineering-for-ai-agents)
- **Blog post (onseok.dev, Mar 2026):** Production RAG systems fail at retrieval ~40% of the time — not at generation. The "lost in the middle" problem means that even when hybrid search retrieves the right chunk, placing it in the middle of a long context window makes it effectively invisible to the model. Q&A-augmented chunking — structuring chunks as question-answer pairs rather than prose — improves both retrieval relevance and the chunk's positional signal when inserted into context. — [onseok.dev](https://onseok.github.io/posts/building-production-rag-system/)
- **Blog post (Lushbinary, 2026):** Context engineering is the discipline covering everything the model sees at inference time: what you include, exclude, compress, position, and how you preserve cache state. The four core strategies are write (author context deliberately), select (filter before insertion), compress (summarize history at turn boundaries), and isolate (keep perception output separate from reasoning input). — [lushbinary.com](https://lushbinary.com/blog/context-engineering-ai-agents-production-guide)

## Gotchas

- **Adding more context rarely helps.** Teams faced with poor agent output add more retrieved chunks, more history, more tool descriptions. This almost always makes it worse by pushing the relevant signal deeper into the middle of the context where the model's attention is weakest.
- **Rerankers can hurt if they change the position of the best chunk.** A reranker that surfaces the correct document at rank 1 but delivers it in a 5,000-token chunk positioned in the middle of context provides worse signal than a smaller, more precisely extracted chunk at the start of context.
- **Cache persistence breaks position assumptions across turns.** What works in a single-turn test fails in a 20-turn agent because the KV cache state from previous turns shifts the effective position of new context elements. Test position contracts under realistic multi-turn scenarios, not in isolated single-turn evaluations.
- **System prompt placement is not free.** Putting a long system prompt at the very start of every turn consumes the most-attended boundary with relatively static information. Smaller, turn-specific dynamic instructions placed at the end of the system prompt (just before the model's generation point) often outperform long preambles at the start.
