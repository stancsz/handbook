# S-246 · Multi-Agent Production Patterns: State, Observability, and Failure Design

A single agent in a notebook is not the same thing as a multi-agent pipeline in production. The gap is not model quality — it is state management, failure propagation, and observability. Teams that survive production learn these the hard way.

## Forces

- **Non-determinism compounds across agent boundaries.** Each LLM call introduces variance. Chained agents multiply it — one agent's degraded output becomes the next agent's degraded input, cascading into system-level failure. — [DevStarsJ, Multi-Agent AI Systems: Architecture Patterns for Building Reliable Agent Pipelines, March 2026](https://devstarsj.github.io/ai/architecture/2026/03/14/multi-agent-ai-architecture-patterns-2026)
- **State management is the unsexy core.** LangGraph agents that work in testing silently loop for 20+ minutes in production, consuming $200+ in API calls before anyone notices. The root cause is almost always implicit state — agents without typed schemas, checkpointing, or recovery paths. — [Markaicode, Production Multi-Agent System with LangGraph, March 2026](https://markaicode.com/langgraph-production-agent)
- **Evals are not optional.** Multiple teams report that prompt tweaks "passed a vibe check" but failed against a structured eval suite. Without evals, you do not know if you are moving the needle. The counter-argument — that LLM-as-critic works — has no empirical support; internal experiments with researchers found LLMs are not reliable critics. — [Hacker News, Principles for production AI agents discussion, citing app.build, 2025](https://news.ycombinator.com/item?id=44712315)
- **Orchestration topology matters more than model choice.** Google's internal "Agent Bake-Off" found distributed multi-agent architectures cut processing time from 1 hour to 10 minutes — a 6× gain — compared to monolithic single-agent approaches. AdaptOrch (2026) research found topology improvements yielded 12–23% gains on SWE-bench, exceeding the gains from swapping underlying models. — [MACGPU, Multi-Agent AI Architecture in Production, June 2026](https://macgpu.com/en/blog/2026-0622-multi-agent-ai-architecture-production-guide.html)

## The move

Build multi-agent pipelines with explicit state contracts, per-step failure recovery, and structured observability from day one — not as an afterthought.

- **Use typed state schemas, not bare dicts.** Define `AgentState` with `TypedDict` for every agent. This enables static analysis, catches schema drift, and makes checkpoint serialization reliable. Bare dicts work for prototypes; they fail silently in production when state shapes change across updates. — [Markaicode, Production Multi-Agent System with LangGraph, March 2026](https://markaicode.com/langgraph-production-agent)
- **Implement checkpointing for any long-running pipeline.** Use `MemorySaver` in development and `RedisCheckpointer` or `PostgresCheckpointer` in production. Checkpoints enable human review, replay from failure points, and graceful resume after crashes — not just recovery, but accountability. — [Markaicode, Production Multi-Agent System with LangGraph, March 2026](https://markaicode.com/langgraph-production-agent)
- **Recover at the step level, not the graph level.** A global `try/except` around an entire graph is useless — you cannot identify which agent failed or what state it left behind. Wrap individual node calls with per-step error handlers that either retry, fall back to a default, or escalate with structured error context. — [DevStarsJ, Multi-Agent AI Systems: Architecture Patterns, March 2026](https://devstarsj.github.io/ai/architecture/2026/03/14/multi-agent-ai-architecture-patterns-2026)
- **Trace every agent turn end-to-end.** LangSmith, Phoenix (by Arize), or custom OpenTelemetry instrumentation — pick one and instrument every node. Without trace correlation across agents, debugging a 5-agent pipeline is archaeology. — [Markaicode, Production Multi-Agent System with LangGraph, March 2026](https://markaicode.com/langgraph-production-agent)
- **Build evals before you need them.** Start with 10–20 golden case datasets from real production failures. Even "vibe check" evals — written down and run consistently — outperform intuition. Calibrate LLM-as-judge against human annotations; do not trust it standalone. — [Hacker News, Principles for production AI agents, 2025](https://news.ycombinator.com/item?id=44712315)
- **Plan token budgets at the graph level.** A 5-agent pipeline with 3 tool calls each = 15+ LLM calls per user request. Track cumulative token spend per pipeline run and fail fast if a threshold is breached. — [DevStarsJ, Multi-Agent AI Systems: Architecture Patterns, March 2026](https://devstarsj.github.io/ai/architecture/2026/03/14/multi-agent-ai-architecture-patterns-2026)

## Gotchas

- **Implicit state is a production liability.** The prototype's bare-dict state that seemed flexible becomes a silent failure mode when the schema drifts across versions or when you need to replay a specific turn. Lock schemas early, version them, and treat schema changes as breaking changes.
- **Multi-agent cost scales superlinearly.** Each additional agent in a pipeline multiplies token costs and latency. Teams that skip per-run token budgets routinely discover $200–$800/month overruns after a few weeks in production. Budget at design time, not retroactively.
- **LLM-as-judge eval is uncalibrated without human ground truth.** Teams that skip human annotation on their eval set find that LLM judges drift in their criteria over time, producing inconsistent scores that mask real regressions. Calibrate against humans on at least 10–15 cases before going autonomous.
- **Sandbox individual agent tool execution.** In multi-agent systems, one agent's malformed tool call can corrupt shared state or exhaust rate limits for all other agents. Isolate tool execution per agent and implement per-agent rate limiting and timeout budgets.
